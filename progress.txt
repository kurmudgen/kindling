PHASE 3.5 SELF REVIEW — PRODUCTION HARDENING
==========================================================================

CURRENT ERROR HANDLING ACROSS THREE TIERS (pre-Phase 3.5):

Tier 1 (tier1.ts): One path, no retries. If Ollama's generate() throws,
the error bubbles up. No timeout, no fallback model.

Tier 2 (tier2.ts): Had secondary API fallback path but only activated
at the tier level — a runtime Ollama hang would block indefinitely.
No retry, no timeout.

Tier 3 (tier3.ts): Routed between local and API with a single try/catch.
Both failing threw "No backend available".

Router (router.ts): generateFromTier() walked UP on failure to more
expensive tiers — wrong direction. Infrastructure failures should
cascade DOWN to cheaper tiers, not up to more expensive ones.

No circuit breaker, no retries, no structured recovery. Not production-grade.

LEARNED.JSON ACCUMULATION (pre-Phase 3.5):

Flat keyed object with no timestamps or session counts. Each new sleep
session OVERWROTE adjustments for touched signals. Stale recommendations
from 20 sessions ago persisted with full weight until a newer session
touched them. No decay — system could not forget.

CONCURRENT QUERIES (pre-Phase 3.5):

Router shared mutable state (confidence, buffer, meta) across queries.
Two concurrent queryDetailed() calls would corrupt each other. Log
writes used fire-and-forget appendFile which could interleave.

SLEEP ANALYST LOG INPUT (pre-Phase 3.5):

Full raw escalation.jsonl parsed and sent to API. No compaction.
50-query sessions would push 8000+ tokens before analysis budget.

  Tool: src/tools/test-decay.ts

MILESTONE 3 — 5-LAYER ERROR RECOVERY: PASS
  src/recovery/recovery.ts implements:
  - Layer 1: Retry with exponential backoff (1s, 2s, 4s) for
    transient errors (ECONNRESET, ETIMEDOUT, fetch failed, timeouts)
  - Layer 2: Tier downgrade — fail DOWN on infrastructure failure
    (not up, which was wrong)
  - Layer 3: Model fallback — fallbackModel per tier in config
  - Layer 4: API fallback — apiFallbackModel via Anthropic
  - Layer 5: Circuit breaker — trips after 3 consecutive degradations,
    logs warning, resets on first clean query
  - Timeout wrapper on every tier call (configurable, default 120s)
  - All tier classes now expose generateWithModel() and
    generateWithApi() hooks for the cascade
  - Recovery events logged to escalation.jsonl via logRecoveryEvent
  - Write queue serializes all log writes across concurrent calls

  Verified: simulated Tier 2 failure (bogus model names) triggered
  cascade through model_fallback → api_fallback, query succeeded
  via Haiku API, 2 recovery events logged. Gate: PASS.
  Tool: src/tools/test-recovery.ts

MILESTONE 4 — LOG COMPACTION: PASS
  src/sleep/compactor.ts three-tier strategy:
  - Tier 1: Strip redundant fields, use 1-2 char key names
  - Tier 2: Collapse runs of 3+ similar events into repetition summary
  - Tier 3: One-line structured session summary header
  - analyst.ts runs compactLog() before API call and logs reduction

  Results on real sessions: 73-83% token reduction
  Synthetic 50-event log: 3874 → 945 tokens (75.6% reduction, well
  under the 2000-token gate target).
  Tool: src/tools/test-compactor.ts

MILESTONE 5 — VCR SESSION REPLAY: PASS
  src/tools/vcr.ts implements:
  - Record mode (KINDLING_VCR=record): writes every routing decision
    to logs/vcr/<uuid>.jsonl with signals, valence, start/final tier,
    confidence, router decision, meta action, recovery event count
  - Privacy: prompt HASH only, never raw prompt text
  - Replay mode (KINDLING_VCR=replay + KINDLING_VCR_FILE=path):
    re-runs each recorded decision through current confidence +
    meta model, diffs against recorded, reports matches and reasons
    for mismatches
  - Router emits VCR entry after every query when recording active

  Verified: 5-query record session, privacy check passed (no raw
  prompts in file), structure check passed, replay produced 5/5

  Tool: src/tools/test-vcr.ts
