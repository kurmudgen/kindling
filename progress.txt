WHAT PHASE 2 LEFT INCOMPLETE OR APPROXIMATED:

1. Router is still batch, not streaming (router.ts:98-103). The full Tier 1
   response is generated, THEN signals are checked once. The spec calls for
   per-token confidence routing via streaming. Phase 2 noted this but did
   not fix it. Acceptable — streaming is Phase 3+ scope.

2. The speculative buffer (speculative.ts) is tested but never actually used
   in the real routing flow. The router generates a complete response from
   each tier. The buffer's fill/confirm/reject lifecycle is only exercised
   in unit tests. The router overrides or replaces tokens wholesale.

3. Attention archaeology signal (signals.ts:148-170) uses skip-bigram and
   trigram repetition heuristics. The spec called for actual attention head
   activation tracking, which Ollama does not expose via API. The heuristic
   is a reasonable approximation — it catches degenerate outputs but cannot
   detect subtle attention anomalies. Documented limitation.

4. Cold concept warming (prewarmer.ts) does prompt prefix injection — runs
   a short "Briefly define: X" through Tier 1. It does NOT inject into KV
   cache (Ollama limitation). The warming effect is limited to keeping the
   model hot in RAM/CPU cache. Not measured whether it actually helps.

5. The sleep analyst (analyst.ts) has NEVER BEEN RUN. No learned.json exists.
   No sleep logs exist. The entire learning system is untested in production.
   It requires ANTHROPIC_API_KEY which was invalid during Phase 1/2 testing.

WHAT THE LEARNING SYSTEMS CURRENTLY DO vs SPEC:

- Sleep analyst: Designed to analyze escalation logs and suggest weight
  adjustments. Code exists but is untested. The API call, JSON parsing,
  learned.json writing, and prewarmer invocation are all implemented but
  never exercised with real data.

- Learned weights: config.ts:82-88 loads learned.json and deep-merges it
  over the profile config. The merge logic works — tested by code review.
  But learned.json has never been created, so the merge has never fired.

- Weight application: signals.ts:183-191 reads getConfig().escalation for
  weights. Since learned.json merges into the escalation section, any
  learned adjustments WOULD affect computeConfidence. The pipeline is
  correctly wired but untested end-to-end.

WHAT learned.json CONTAINS: Does not exist. No sleep session has ever run.
