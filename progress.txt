PHASE 3.5 SELF REVIEW — PRODUCTION HARDENING
==========================================================================

CURRENT ERROR HANDLING ACROSS THREE TIERS:

Tier 1 (tier1.ts): One path, no retries. If Ollama's generate() throws,
the error bubbles up to the router. No timeout wrapping. No fallback
model. If the model isn't pulled, warmup() throws a user-friendly error
but generate() just dies.

Tier 2 (tier2.ts): Has a secondary path (API fallback via
generateAPI()) but only activates when isOllamaModelAvailable() returns
false at call time. If Ollama is up but the model is slow/stuck, it
hangs indefinitely. No retry. No timeout. The API fallback itself has
no retry — a 500 from Anthropic kills the query.

Tier 3 (tier3.ts): Routes between Tier3Local and API. The fallback
from local to API has a single try/catch but no retry. If BOTH fail,
throws "Tier 3: No backend available". The router catches this and
tries the next lower tier, but that's ad-hoc not structured.

Router (router.ts:255-271): The generateFromTier() method has a loop
that walks upward from the requested tier. If a tier fails, it tries
the NEXT HIGHER tier — this is wrong for failure recovery. Failure
should cascade DOWN to cheaper tiers, not up to more expensive ones.
The current behavior means a Tier 1 failure tries Tier 2, which
defeats the "low burn" promise and wastes budget on infrastructure
failures rather than genuine escalation needs.

No circuit breaker. No retry with backoff. No structured recovery
cascade. This is not production-grade.

LEARNED.JSON ACCUMULATION OVER TIME:

Current learned.json schema:
  { "escalation": { "escalationThreshold": 0.36, ... } }

Flat keyed object. No timestamps. No session counts. No decay.
When the analyst runs a new session, applyAdjustments() reads the
existing learned.json, SETS each signal's new value (overwrite,
not blend), and writes back. This means:

1. The most recent session always wins completely — there's no
   smoothing across sessions.
2. A stale recommendation from 20 sessions ago has the same weight
   as one from last session (until a newer session overwrites it).
3. If signal X was adjusted 10 sessions ago but no session has
   touched it since, it sits in learned.json forever at its original
   learned value, never decaying back toward defaults even if usage
   patterns have changed.

This is a meaningful hardening gap.

CONCURRENT QUERIES:

The router has shared mutable state:
- this.confidence (ConfidenceAggregator) — has internal
  deescalationCounter state, reset() is called per query.
- this.buffer (SpeculativeBuffer) — reset() per query.
- this.meta (MetaConfidenceModel) — shared history array.

If two queryDetailed() calls happen concurrently, they will BOTH
call reset() on confidence and buffer, step on each other's state,
and produce corrupted routing decisions.

The module-level _router singleton at line 275 means the same Router
instance is reused across all query() calls. There's no isolation.

The logger writes are already async (logEscalation uses fire-and-
forget appendFile), but if two writes happen in the same event loop
tick they can potentially interleave.

SLEEP ANALYST LOG INPUT:

Currently analyst.ts reads the ENTIRE escalation.jsonl, filters to
the current sessionId, parses all events, and sends the full array
in the payload. No compaction. For a benchmark run with 55 queries
and 20 escalations, that's 20 full event objects plus the prior
sessions context — easily 8000+ tokens of input before the response
budget of 8192 even kicks in.

The good news: escalation.jsonl only contains ESCALATION events,
not every query. So the raw log is already partially compacted.
But medium-scale sessions will still blow the context budget.

WHAT I WOULD HARDEN BEFORE CALLING THIS PRODUCTION-READY:

1. Generate-number concurrency isolation (router per-query state)
2. Proper failure cascade (fail DOWN not UP on infrastructure errors)
3. Retry with backoff for transient Ollama/API failures
4. Circuit breaker for sustained infrastructure degradation
5. Learned weight decay so the system forgets stale patterns
6. Log compaction before analyst calls
7. Fallback model per tier in config (not just API fallback)
8. Dream task scheduler that runs on idle without user intervention
9. VCR replay tool for debugging routing decisions after-the-fact
10. Concurrent query isolation in the router

Phase 3.5 covers items 1, 2, 3, 4, 5, 6, 8, 9, 10. Item 7 is covered
as part of the 5-layer recovery milestone.
